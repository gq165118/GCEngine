# GCEngine 项目上手与架构复盘

> 面向“快速读懂代码”的导览文档。重点说明：模块边界、关键数据流、当前代码健康度、建议上手路径。

## 1. 项目定位（一句话）

这是一个偏教学/实验性质的 C++ OpenGL 渲染引擎骨架：
- 以 `Object3D` 场景树为核心；
- 以 `Geometry + Material + Texture` 描述可渲染资源；
- 以 `Driver*` 后端层负责 GPU 资源同步和状态缓存；
- 通过事件系统做 CPU 资源析构到 GPU 资源释放的联动。

## 2. 顶层目录和构建方式

- 顶层 CMake 只做三件事：设置 C++17、引三方头库、进入 `ff/` 子目录构建。`ff` 目录被整体编译为 `ff_lib` 静态/动态库（取决于生成器）。
- 三方依赖在 `thirdParty/`：GLFW、GLAD、GLM、Assimp、stb。

## 3. 模块架构总览（按职责）

### 3.1 基础层（global/tools）
- `global/base.h`：基础类型与常用头文件聚合（`ID`、`Range`、`KeyBoardState` 等）。
- `tools/identity.*`：全局递增 ID 生成器，几乎所有资源对象都依赖它分配唯一 ID。
- `global/constant.h`：输入映射、OpenGL 枚举封装与转换函数（`toGL(...)`、`toDataType<T>()` 等）。
- `global/eventDispatcher.*`：单例事件总线，支持注册成员函数监听器并按事件名分发。

### 3.2 场景与对象层（core/objects/scene/camera）
- `Object3D`：全场景共同基类。负责 TRS、本地矩阵/世界矩阵、父子层级、法线矩阵与模型视图矩阵。
- `Scene`：继承 `Object3D`，作为场景根节点，增加全局覆盖材质、天空盒背景。
- `RenderableObject`：将 `Geometry` 与 `Material` 绑定到可渲染实体，并支持渲染前回调。
- `Mesh`：最常见渲染对象类型，本质是 `RenderableObject` 的语义化子类。
- `Camera`：继承 `Object3D`，多维护一个 `worldMatrixInverse`（view 矩阵）与 `projectionMatrix`。
- `PerspectiveCamera/OrthographicCamera`：分别实现透视/正交投影矩阵更新。

### 3.3 资源层（geometry/attribute/material/texture）
- `Attribute<T>`：顶点属性模板容器（位置、法线、索引等），记录更新标记与局部更新范围。
- `Geometry`：管理属性字典与索引属性，提供包围盒/包围球计算。
- `Material`：渲染状态与贴图引用集合（混合、深度、面剔除、纹理）。
- `Texture/CubeTexture/DepthTexture`：纹理参数和数据描述。

### 3.4 渲染后端层（render/driver）
- `DriverAttributes`：将 `Attribute` 映射到 VBO/EBO，创建、更新、销毁 GPU Buffer。
- `DriverGeometries`：跟踪 geometry 生命周期，触发 attributes 同步，维护 geometry 统计。
- `DriverBindingStates`：维护 VAO 绑定状态（接口存在，但实现细节仍较少）。
- `DriverInfo`：记录帧统计（draw call、triangle 等）。
- 另有 `DriverObjects/DriverUniforms/DriverRenderList` 等文件，框架存在但部分代码明显未完成。

## 4. 关键运行链路（你要先建立这个心智模型）

### 链路 A：场景树变换传播
1. 业务侧对 `Object3D` 调整 TRS 或直接设 localMatrix。
2. 调 `updateWorldMatrix(updateParent, updateChildren)`：
   - 可选择先递归更新父节点；
   - 计算当前节点 worldMatrix；
   - 可选择递归更新子节点。
3. 渲染前结合 camera viewMatrix 计算 modelViewMatrix，再推导 normalMatrix。

### 链路 B：Geometry/Attribute 到 GPU
1. CPU 侧创建 `Geometry` 并填充 `Attribute`。
2. `DriverGeometries::update(geometry)` 遍历 geometry 所有 attribute。
3. `DriverAttributes::update(...)`：
   - 首次：`glGenBuffers + glBufferData`；
   - 增量：依据 `needUpdate / updateRange` 走 `glBufferSubData` 或全量 `glBufferData`。
4. Attribute/Geometry 析构后会抛事件，Driver 层监听后清理缓存记录。

### 链路 C：相机矩阵
1. `Camera::updateWorldMatrix()` 调父类更新世界矩阵。
2. 立刻计算逆矩阵作为 viewMatrix。
3. 子类相机按参数更新 projectionMatrix（透视/正交）。

## 5. 当前代码健康度（实话实说）

这个仓库“架构主干是清晰的”，但“落地代码尚未收敛到可稳定编译运行”的状态也很明显：

- 命名不一致问题较多（同一语义有 `m_target/mTarget`、`m_pUserData/mpUserData` 等混用）。
- `Attribute<T>` 中有明显拼写和变量名错误（如 `carete`、`iiemSize`、`mData`、`x`），会直接导致编译失败。
- `DriverUniforms` 等文件存在截断/未完成内容，属于骨架阶段。
- 注释完整度高于实现完整度：文档化做得比较积极，但代码一致性还需一轮系统清理。

换句话说：**它更像“正在搭建中的引擎内核”，而不是“可直接用于业务”的成熟库**。

## 6. 建议你按这个顺序上手（高效）

1. **先读场景主干**：`Object3D -> Camera -> Scene -> RenderableObject/Mesh`。
2. **再读资源主干**：`Attribute -> Geometry -> Material/Texture`。
3. **最后读后端同步**：`DriverAttributes -> DriverGeometries -> DriverInfo`。
4. **暂缓深挖**：`DriverUniforms`、shader chunk 复杂组合逻辑（等主干稳定后再看）。

你会发现：前两步决定“引擎怎么描述世界”，第三步决定“如何把世界提交给 GPU”。

## 7. 推荐的第一周阅读清单

- Day 1：`Object3D`（变换 + 层级）
- Day 2：`Camera`（view/projection）
- Day 3：`Geometry/Attribute`（顶点数据结构）
- Day 4：`Material/Texture`（渲染状态）
- Day 5：`DriverAttributes/DriverGeometries`（CPU->GPU）

## 8. 如果你准备“边读边修”

建议从“最低风险、收益最高”的治理开始：

1. 做一次**命名一致性修复**（target/userData 字段名统一）。
2. 修复 `Attribute<T>` 的编译错误与拼写错误。
3. 补最小可运行 demo（单三角形 + 单相机 + 单材质），作为回归基线。
4. 给 Driver 层加 smoke test（至少保证资源创建/销毁链路不崩）。
